# Antigravity Integration Guide

**Date:** 2026-06-05  
**Status:** **ACTIVE / FULLY RESOLVED**

## Summary

The local Antigravity CLI backend (`gemini-3.5-flash` and `gemini-3.5-pro`) is fully wired and integrated with the multi-agent system. This allows the supervisor and its worker pools to complete tasks using your paid Antigravity subscription, resulting in **zero API charges**.

## How It Works (The Architecture)

```
┌────────────────────────────────────────────────────────────────────────┐
│                              VS Code / IDE                             │
│                                                                        │
│   ┌────────────────────────────────────────────────────────────────┐   │
│   │  Integrated Terminal Task (inherits CSRF token + LS port)      │   │
│   │  $ ./bin/antigravity-bridge.exe                                │   │
│   │                                                                │   │
│   │  ┌──────────────────────────────────────────────────────────┐  │   │
│   │  │                Bridge Server (Go)                        │  │   │
│   │  │  1. Receives HTTP request from supervisor                │  │   │
│   │  │  2. Triggers agentapi.bat new-conversation               │  │   │
│   │  │  3. Parses conversationId from JSON metadata             │  │   │
│   │  │  4. Polls transcript.jsonl for completion                │  │   │
│   │  │  5. Returns final PLANNER_RESPONSE text                  │  │   │
│   │  └────────────────────────┬─────────────────────────────────┘  │   │
│   │                           │                                    │   │
│   │                           ▼                                    │   │
│   │              agentapi.bat (authenticated CLI)                  │   │
│   └───────────────────────────┬────────────────────────────────────┘   │
└───────────────────────────────┼────────────────────────────────────────┘
                                │ HTTP POST (REST)
                                ▼
                   http://localhost:8765/prompt
                                ▲
                                │
    ┌───────────────────────────┴────────────────────────────────────┐
    │                     Supervisor.exe                             │
    │  - Spawns worker threads (gemini-flash, gemini-pro)            │
    │  - Workers make API calls via LocalAntigravityBackend          │
    └────────────────────────────────────────────────────────────────┘
```

### 1. Bypassing the CSRF Authentication Blocker
Programmatically executing `agentapi.bat` from external shells fails with a `missing CSRF token` error. To resolve this:
- **Inside the integrated terminal:** Antigravity automatically injects `ANTIGRAVITY_CSRF_TOKEN` and `ANTIGRAVITY_LS_ADDRESS`.
- **Outside the integrated terminal (e.g. Claude Code or external shells):** The bridge server checks the directory structure for a `.env.antigravity` file containing the active session credentials. This file is automatically generated/updated by the Antigravity agent when active:
  ```
  ANTIGRAVITY_CSRF_TOKEN=4281ea1e-ad41-48b1-b74e-c67962479c53
  ANTIGRAVITY_LS_ADDRESS=localhost:53137
  ANTIGRAVITY_PROJECT_ID=4d727caf-f69d-489c-805f-3e94d98fdd79
  ```
  If these environment variables are missing from the shell, the bridge server automatically loads them from the `.env.antigravity` file, allowing it to authenticate with the language server and resolve project configurations from *any* terminal session on the machine.

### 2. Resolving Asynchronous Responses
The `agentapi.bat new-conversation` command triggers a background conversation and returns immediately with JSON confirming creation:
```json
{
  "response": {
    "newConversation": {
      "prompt": "...",
      "conversationId": "ede0f6a6-af25-4523-b1d8-c4529985f26c"
    }
  }
}
```
The bridge server parses this metadata, extracts the `conversationId`, and enters a polling loop. It reads the local transcript log file:
`~/.gemini/antigravity/brain/<conversationId>/.system_generated/logs/transcript.jsonl`

It scans the log lines for a step where:
- `source` is `"MODEL"`
- `type` is `"PLANNER_RESPONSE"`
- `status` is `"DONE"` (success) or `"ERROR"` (failure)

Once found, it extracts the `content` field (the model's generated text) and returns it as the HTTP response.

## Setup & Running Guide

### Step 1: Start the Bridge Server
The bridge server must run in the integrated terminal to authenticate.

**Option A (VS Code Task - Recommended):**
1. Open the project in VS Code.
2. Press `Ctrl+Shift+P` (or `Cmd+Shift+P` on Mac).
3. Type **Tasks: Run Task** and press enter.
4. Select **Start Antigravity Bridge**.

**Option B (Manual Terminal):**
1. Open a terminal panel inside VS Code (`` Ctrl+` ``).
2. Run the helper script:
   ```powershell
   cd multi-agent-system
   .\run-bridge.bat
   ```

### Step 2: Start the Supervisor
Once the bridge server is running, start the supervisor. It can run in any terminal.

**Option A (VS Code Task - Recommended):**
1. Open the command palette (`Ctrl+Shift+P`).
2. Type **Tasks: Run Task** and select **Start Multi-Agent Supervisor**.

**Option B (Manual Terminal):**
1. Open a separate shell (standard PowerShell, cmd, bash, etc.).
2. Run the compiled binary:
   ```powershell
   cd multi-agent-system
   .\supervisor.exe
   ```

## Verifying the Setup

You can verify the bridge server independently from the supervisor.

Send a POST request using PowerShell's `Invoke-RestMethod`:
```powershell
Invoke-RestMethod -Uri "http://localhost:8765/prompt" -Method Post -ContentType "application/json" -Body '{"prompt":"What is 2+2?","model":"flash"}'
```

**Expected Output:**
```
response       
--------       
2 + 2 is **4**.
```

When checking the bridge server's log output, you should see:
```
Received prompt request: model=flash, prompt_length=12
Created conversation <uuid>. Waiting for model response...
Success: response_length=15
```
